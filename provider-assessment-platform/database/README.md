# قاعدة بيانات منصة التقييم المهني

## الحالة

مخطط تصميم وتجربة معزول. لم يطبق على قاعدة بيانات إنتاجية، ولا يحتوي على بيانات أشخاص حقيقيين.

## المنصة المستهدفة

- PostgreSQL 15 أو أحدث.
- اتصال TLS إلزامي.
- دور ترحيل منفصل عن أدوار التطبيق.
- تخزين مفاتيح التشفير خارج قاعدة البيانات.
- نسخ احتياطي مشفر واختبار استعادة دوري.

## ترتيب الترحيلات

1. `postgresql.v0.1.0.sql` — الجداول والقيود الأساسية وسجل التدقيق وRLS الأولي.
2. `postgresql.v0.1.1-security-hardening.sql` — تقييد الوصول بتكليف الحالة وتأمين العارضات.
3. `postgresql.v0.1.2-cross-tenant-integrity.sql` — مفاتيح أجنبية مركبة تمنع الربط بين المؤسسات والحالات وتشديد إدخال التدقيق.

يجب تنفيذها بالترتيب داخل بيئة فارغة. لا يجوز تجاوز ترحيل أو تعديل ملف سبق تطبيقه؛ ينشأ ملف ترحيل جديد.

## حدود البيانات

### مخزن الهوية

الأسماء وأرقام الهوية والعناوين وأرقام الاتصال وبيانات ولي الأمر تبقى في مخزن هوية منفصل مشفر. جدول `cases` يحتفظ فقط بـ`identity_vault_reference` غير قابل للاستخدام خارج الخدمة المخولة.

### مخزن العمل السريري

يحتوي على:

- حالة الملف والموافقة والإحالة.
- المصادر والسلامة والمسار.
- خطة التقييم والجلسات والانحرافات.
- مراجعة الفريق وإصدارات التقارير.
- سجل التدقيق.

### حدود الناشر

بنود الاختبار وتعليماته المحمية ومفاتيح التصحيح وجداول المعايير لا تخزن في هذا المخطط. `result_reference` مرجع معتم فقط إلى تكامل مرخص أو نتيجة خارجية مصرح باستيرادها.

## سياق الجلسة الإلزامي

قبل أي استعلام للمستخدم يضبط التطبيق داخل المعاملة:

```sql
SELECT set_config('app.institution_id', :institution_id, true);
SELECT set_config('app.provider_id', :provider_id, true);
SELECT set_config('app.audit_scope', :audit_scope, true);
```

- `audit_scope` فارغ في الاستخدام المعتاد.
- القيمة `institution` تمنح فقط لدور تدقيق مستقل وبعد تسجيل السبب.
- إذا لم يضبط السياق تفشل سياسات RLS بصورة مغلقة ولا تعيد صفوفًا.
- لا يسمح للتطبيق بتلقي هذه القيم من الطلب مباشرة؛ تستخرج من هوية موثقة ومطالبات خادم الهوية.

## الكتابة والتصحيح

- السجلات السريرية والأدلة والتقارير والتدقيق غير قابلة للتعديل أو الحذف.
- التصحيح ينشئ إصدارًا أو حدثًا جديدًا مرتبطًا بالأصل.
- الحالة الوحيدة القابلة للتحديث تحمل رقم نسخة يزداد آليًا.
- API يحتاج `If-Match` لمنع فقدان تحديث متزامن.
- التقرير الموقّع لا يتغير؛ التصحيح إصدار جديد ويصبح السابق `superseded` أو `withdrawn` وفق السياسة.

## سجل التدقيق

يخزن كل حدث بصمة `event_hash` وبصمة الحدث السابق عند توفرها. يجب أن تنفذ خدمة التدقيق خارج المعاملة السريرية تحققًا دوريًا من السلسلة وتنسخها إلى مخزن غير قابل للتغيير.

لا يجوز اعتبار البصمة وحدها توقيعًا قانونيًا؛ يلزم نظام إدارة مفاتيح وتوقيع وختم زمني عند اشتراط القانون.

## التحقق قبل أي تجربة

- تشغيل الترحيلات على PostgreSQL 15 و16 في CI.
- اختبار RLS بدورين من المؤسسة نفسها ودور من مؤسسة أخرى.
- اختبار عدم الوصول إلى حالة غير مكلف بها المستخدم.
- اختبار جميع المفاتيح الأجنبية المركبة بمحاولات ربط عبر حالة أو مؤسسة أخرى.
- اختبار عدم إمكانية تعديل الجداول غير القابلة للتغيير.
- اختبار استعادة نسخة احتياطية مشفرة.
- مراجعة خطة الاحتفاظ والحذف والنسخ القانونية.
- اختبار الأداء مع بيانات صناعية بالحجم المتوقع.

## محظورات

- تشغيل الترحيلات يدويًا على الإنتاج دون سجل إصدار ومراجعة.
- منح أدوار التطبيق `BYPASSRLS` أو ملكية الجداول.
- استخدام دور الترحيل في الخدمة اليومية.
- تخزين أسرار أو مفاتيح تشفير أو مواد اختبار محمية في الجداول.
- تعطيل RLS لمعالجة مشكلة تطبيقية.
- حذف حدث تدقيق أو تقرير موقّع لتصحيح خطأ.
