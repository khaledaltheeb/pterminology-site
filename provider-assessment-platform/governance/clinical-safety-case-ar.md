# حجة السلامة السريرية لمنصة التقييم المهني

> هذه الوثيقة تجمع الادعاءات والأدلة والاختبارات التي يجب توافرها قبل الاستخدام السريري. لا تعني صياغة الادعاء أنه مثبت؛ الحالة الافتراضية لكل ادعاء هي **غير مثبت** حتى ربطه بدليل مستقل ناجح.

## 1. الادعاء الأعلى

**C0 — المنصة آمنة بما يكفي ضمن غرض محدد ونطاق مؤسسات وأدوات ومستخدمين معتمدين، ولا تسمح بأن يحل النظام محل الحكم المهني أو أن يصدر نتيجة عالية الأثر من بيانات ناقصة أو أداة منفردة.**

حالة الادعاء: `NOT_PROVEN`

لا يتحول إلى `SUPPORTED` قبل اكتمال جميع الادعاءات الفرعية الحرجة وعدم وجود خطر متبقٍ غير مقبول.

## 2. نطاق الغرض المقصود

يجب استكماله قبل التقييم:

| البند | القيمة |
|---|---|
| المستخدمون | مقدمو خدمة ومراجعون معتمدون داخل مؤسسات متعاقدة |
| الأشخاص | `PENDING: الأعمار والحالات واللغات` |
| الدول | `PENDING` |
| الوظائف | استقبال، تخطيط، جمع أدلة، توجيه، مراجعة، تقرير، متابعة |
| الوظائف المستبعدة | تشخيص آلي، أهلية آلية، وصف دواء، استبدال الحكم المهني |
| الأدوات المفعلة | `PENDING: أداة وإصدار ولغة ومؤسسة` |
| نموذج النشر | `PENDING` |
| الاعتماد التنظيمي | غير محدد وغير حاصل حاليًا |

أي استخدام خارج هذا النطاق يبطل حجة السلامة.

## 3. بنية الادعاءات

```text
C0 السلامة الكلية
├── C1 المستخدم والمؤسسة والأداة مخولون
├── C2 الحالة والهوية والموافقة صحيحة
├── C3 المخاطر العاجلة توقف المسار الروتيني
├── C4 الأدلة كافية وصالحة ومتعددة المصادر
├── C5 قواعد الانتقال صحيحة وقابلة للتفسير
├── C6 التقرير نتيجة مراجعة بشرية ومقيد بالأدلة
├── C7 الخصوصية والأمن يمنعان الوصول والتلاعب غير المصرح
├── C8 الإتاحة والتكييف لا يفسدان صلاحية القياس
├── C9 الأعطال والحوادث تكتشف وتحتوى ويصحح أثرها
└── C10 الأداء والإنصاف يراقبان بعد الإطلاق
```

## 4. سجل الادعاءات والأدلة

### C1 — التفويض المهني والمؤسسي

**الادعاء:** لا يستطيع المستخدم بدء عملية مقيدة إلا إذا كانت المؤسسة والمستخدم والأداة والعملية والحالة جميعها ضمن تفويض صالح لحظة التنفيذ.

**الحجج المطلوبة**

- هوية المستخدم موثقة وMFA مفعّل.
- العلاقة بالمؤسسة فعالة.
- الترخيص المهني والتدريب غير منتهيين.
- المؤسسة معتمدة للعملية.
- الأداة وإصدارها ولغتها وحقها الرقمي مفعلة.
- المستخدم مكلف بالحالة.
- الموافقة تشمل العملية.
- لا يوجد تعليق أو خطر يمنع الجلسة.

**الأدلة**

| الدليل | الحالة | المرجع |
|---|---|---|
| مخطط صلاحيات fail-closed | موجود كمسودة | `engine/access.py` |
| سجل الأدوات والتراخيص | موجود كمسودة | `registry/assessment-catalog.v0.1.0.json` |
| مخطط قاعدة بيانات للتفويض | موجود كمسودة | `database/postgresql.v0.2.0.sql` |
| اختبارات انتهاء الترخيص والتدريب | جزئية | `tests/test_access.py` |
| اختبار تكامل خادمي فعلي | غير موجود | `PENDING` |
| مراجعة حقوق ناشر لكل أداة | غير موجودة | `PENDING` |

**قرار الادعاء:** `PARTIALLY_SUPPORTED — NOT RELEASEABLE`

### C2 — صحة الحالة والهوية والموافقة

**الادعاء:** يمنع النظام معالجة الحالة الخطأ أو العمل خارج نطاق الموافقة.

**الحجج المطلوبة**

- الهوية منفصلة عن قاعدة العمل السريري.
- جميع السجلات مرتبطة بالمؤسسة والحالة بمفاتيح مركبة.
- يظهر معرّف الحالة بوضوح في العمليات عالية الخطر.
- الموافقة ذات إصدارات ولا تعدل بصمت.
- السحب يوقف العمليات التي تعتمد عليها مع الحفاظ على الالتزامات القانونية.
- الاستيراد والتقرير والتوقيع تتحقق من الحالة والإصدار.

**الأدلة**

| الدليل | الحالة | المرجع |
|---|---|---|
| مرجع مخزن هوية منفصل | مصمم | `database/postgresql.v0.2.0.sql` |
| مفاتيح مؤسسة وحالة مركبة | مصممة | `database/postgresql.v0.2.0.sql` |
| موافقة ذات إصدارات وغير قابلة للتعديل | مصممة | `consent_versions` |
| اختبارات ربط عبر مؤسسة أخرى | موجودة صناعيًا | `database/tests/rls-smoke-v2.sql` |
| اختبار التباس نافذتين/حالتين | غير موجود | `PENDING` |
| سياسة سحب الموافقة بحسب القانون | غير معتمدة | `PENDING` |

**قرار الادعاء:** `PARTIALLY_SUPPORTED — LEGAL REVIEW REQUIRED`

### C3 — السلامة العاجلة

**الادعاء:** ظهور خطر عاجل أو طارئ يوقف جميع الانتقالات الروتينية ويطلق مسارًا محليًا واضحًا مع تسليم موثق.

**الحجج المطلوبة**

- فحص السلامة يحدث في نقاط محددة وعند معلومة جديدة.
- مستوى `urgent` أو `emergency` يتقدم على بقية القواعد.
- الحالة تنتقل إلى `safety_hold`.
- لا يعود المسار إلا بمراجعة مخولة.
- بيانات الاتصال المحلية مختبرة.
- يوجد دليل استلام الجهة المحال إليها.

**الأدلة**

| الدليل | الحالة | المرجع |
|---|---|---|
| أولوية سلامة في المحرك | موجودة ومختبرة | `engine/runtime.py`, `tests/test_all_pathways.py` |
| مشغل قاعدة بيانات يوقف الحالة | موجود كمسودة | `safety_event_blocks_case` |
| اختبار صناعي لحدث عاجل | موجود | `database/tests/rls-smoke-v2.sql` |
| سياسة سلامة عامة | موجودة | `policies/safety-and-escalation.md` |
| خدمات الطوارئ المحلية لكل مركز | غير موجودة | `PENDING` |
| تمرين تسليم واستلام | غير منفذ | `PENDING` |

**قرار الادعاء:** `PARTIALLY_SUPPORTED — LOCAL CONFIGURATION REQUIRED`

### C4 — كفاية وصلاحية الأدلة

**الادعاء:** لا يصل النظام إلى مراجعة نهائية أو تقرير عندما تكون الأدلة ناقصة أو غير صالحة أو تعتمد على أداة واحدة.

**الحجج المطلوبة**

- البيانات المفقودة `UNKNOWN` وليست `false`.
- كل مسار يحدد الحد الأدنى للمصادر والمجالات.
- النتيجة غير الصالحة تبقى منفصلة عن الدرجة.
- التعارض الجوهري يوجه إلى مراجعة.
- اللغة والسمع والبصر والحركة والتعليم تفحص عند صلتها.
- لا تعرض درجة معيارية خارج شروط النسخة والمعايير.

**الأدلة**

| الدليل | الحالة | المرجع |
|---|---|---|
| منطق ثلاثي | موجود ومختبر | `engine/rules.py`, `tests/test_rules.py` |
| منع الانتقال من جمع الأدلة مباشرة للتقرير | موجود | `validate_cross_references.py` |
| ستة مسارات متعددة الأدلة | موجودة كمسودات | `pathways/*.json` |
| سياسات اللغة والتكييف | موجودة | `policies/multilingual-assessment.md`, `policies/access-accommodations.md` |
| اختبارات أدوات مرجعية فعلية | غير موجودة | `PENDING` |
| تحقق سيكومتري محلي | غير موجود | `PENDING` |

**قرار الادعاء:** `STRUCTURALLY_SUPPORTED — CLINICAL EVIDENCE PENDING`

### C5 — صحة وقابلية تفسير قواعد الانتقال

**الادعاء:** كل انتقال له شرط محدد ومصدر وإصدار وتفسير، ويمكن إعادة إنتاجه ومراجعته.

**الحجج المطلوبة**

- القواعد حتمية وليست مولدة لحظيًا.
- تعريف المسار له إصدار وبصمة ومصادر.
- يحتفظ القرار بالحقيقة والحقول المفقودة والتفسير.
- الانتقالات عالية الأثر تحتاج تأكيدًا بشريًا.
- القاعدة المنتهية أو المعلقة لا تستخدم في حالة جديدة.
- الحالات الجارية تحتفظ بالإصدار المستخدم.

**الأدلة**

| الدليل | الحالة | المرجع |
|---|---|---|
| محرك حتمي | موجود | `engine/runtime.py` |
| مخطط مسار صارم | موجود | `schemas/pathway-definition.schema.json` |
| مصادر وسجل تغيير | موجود في المسارات | `pathways/*.json` |
| اختبارات مستودع وترابط | موجودة | `validate_repository.py`, `validate_cross_references.py` |
| توقيع واعتماد علمي للمسارات | غير موجود | `PENDING` |
| مقارنة مستقلة مع الإرشادات | جزئية | `PENDING` |

**قرار الادعاء:** `TECHNICALLY_SUPPORTED — SCIENTIFIC SIGN-OFF PENDING`

### C6 — سلامة التقرير والتوقيع

**الادعاء:** التقرير مسودة حتى يراجعه ويوقعه مختص مخول، ولا يتغير التقرير الموقع بصمت.

**الحجج المطلوبة**

- إنشاء المسودة منفصل عن التوقيع.
- المراجع يرى الأدلة المؤيدة وغير المؤيدة والقيود.
- التوقيع يحتاج دورًا ومرجع ترخيص وبصمة محتوى.
- التصحيح إصدار جديد مرتبط بالأصل.
- التقرير لا يكشف مواد اختبار محمية.
- الذكاء الاصطناعي، إن استخدم، لا ينتج نصًا موقعًا مباشرة.

**الأدلة**

| الدليل | الحالة | المرجع |
|---|---|---|
| عقد API لمسودة وتوقيع منفصلين | موجود كمسودة | `api/openapi.v0.2.0.json` |
| تقرير ذو إصدارات وبصمة | موجود كتصميم | `report_versions` |
| منع تعديل التقرير | موجود | trigger `report_versions_immutable` |
| مراجعة بشرية في المسارات | موجودة | عقد `multidisciplinary-review` و`professional-report` |
| خدمة توقيع حقيقية وإدارة مفاتيح | غير موجودة | `PENDING` |
| اختبار تقرير كامل من البداية للنهاية | غير موجود | `PENDING` |

**قرار الادعاء:** `DESIGNED — NOT IMPLEMENTED END TO END`

### C7 — الأمن والخصوصية

**الحجج المطلوبة**

- عزل المؤسسة والحالة عبر الخادم وقاعدة البيانات.
- أدوار التطبيق بلا `BYPASSRLS` ولا ملكية الجداول.
- تشفير ومفاتيح منفصلة.
- سجلات غير قابلة للتعديل ومحاولات الرفض مسجلة.
- تصدير محدود ومشفّر ومراقب.
- لا بيانات حقيقية في CI أو التطوير.
- حقوق الأشخاص والاحتفاظ والحذف محددة.

### C8 — الإتاحة وصلاحية التكييف

**الادعاء:** يستطيع الأشخاص والمستخدمون الوصول إلى المنصة، ولا يغير التكييف معنى الأداة دون توثيق أثره.

**الحجج المطلوبة**

- واجهة RTL ولوحة مفاتيح وقارئ شاشة وتباين وتقليل حركة.
- التكييفات مرتبطة بدليل الأداة.
- أي انحراف يسجل مع أثر الصلاحية.
- اللغة ووسيلة التواصل والمترجم موثقة.
- عدم صلاحية الدرجة تؤدي إلى تفسير محدود لا إلى درجة زائفة.

- القدرة على تعليق مؤسسة أو أداة أو مسار.
- تعيين الحالات والتقارير التي استخدمت إصدارًا متأثرًا.
- إصدار تصحيح أو سحب دون حذف الأصل.

### C10 — المراقبة والإنصاف بعد الإطلاق

**الادعاء:** تكتشف المنصة الانحرافات والأضرار والتفاوت غير المبرر وتعلق الوظائف عند الحاجة.

**المؤشرات المطلوبة**

- نسبة النتائج غير الصالحة والمقيدة حسب الأداة والمركز والمستخدم.
- التعارض بين المصادر واللغات والبيئات.
- حالات المسار المتوقفة ونقص الأدلة.
- التصحيحات وسحب التقارير والشكاوى.
- اختلاف التوصيات والنتائج حسب الفئات ذات الصلة دون كشف الهوية.
- تأخر الإحالة والتقييم والتقرير.
- الأدوات أو الحسابات أو التراخيص المنتهية.
- الحوادث والوصول المرفوض والتصدير.

**الضوابط**

- حدود تنبيه وتعليق مسبقة.
- مراجعة بشرية لا تعتمد على مؤشر واحد.
- منع استخدام مؤشرات الإنتاجية للضغط نحو تقليل مدة التقييم.
- مشاركة الأشخاص ذوي الإعاقة والأسر في مراجعة الأثر.

**الأدلة الحالية:** غير منفذة.

**قرار الادعاء:** `NOT SUPPORTED`

## 6. بوابة إطلاق السلامة

لا يسمح بالانتقال من بيانات صناعية إلى تجربة ميدانية ما لم يتحقق:

- [ ] تحديد الغرض والسوق والفئات والأدوات بصورة نهائية.
- [ ] توقيع مالك سريري ومراجع سيكومتري وقانوني وأمني.
- [ ] نجاح الاختبارات التقنية على الإصدار المحدد.
- [ ] تفعيل مؤسسة نموذجية واجتياز اختبار القبول.
- [ ] إعداد السلامة المحلية وتمرينها.
- [ ] اعتماد DPIA وخطة الحادث.
- [ ] حقوق واضحة لكل أداة وتكامل.
- [ ] اختبار إتاحة بمستخدمين فعليين.
- [ ] خطة تجربة ميدانية وموافقة أخلاقية عند الحاجة.
- [ ] معايير إيقاف ومراقبة وتعويض أو تصحيح الضرر.

ولا يسمح بالإطلاق الواسع قبل:

- [ ] نجاح التجربة الميدانية وتحليل الحوادث والانحرافات.
- [ ] تحقق محلي للنسخ العربية والأدوات المؤثرة.
