# نموذج التهديد لمنصة التقييم المهني

> الحالة: مسودة هندسية مستقلة. لا تحل محل اختبار اختراق، أو مراجعة حماية بيانات، أو تقييم تنظيمي، أو مراجعة سلامة سريرية.

## 1. نطاق النموذج

يغطي هذا النموذج:

- حسابات المؤسسات ومقدمي الخدمة.
- ملفات الحالات والموافقات والإحالات.
- محرك المسارات وقواعد الانتقال.
- سجل الأدوات والتراخيص والتدريب.
- جلسات التطبيق واستيراد النتائج.
- مراجعة الفريق والتقارير والتوقيع.
- سجل التدقيق والتكاملات الخارجية.
- النسخ الاحتياطي والاستعادة والمراقبة.

لا يغطي محتوى الأدوات التجارية داخل أنظمة الناشرين؛ يجب إعداد نموذج تهديد مستقل لكل تكامل ناشر.

## 2. الأصول التي يجب حمايتها

| الأصل | أثر التسريب أو التلاعب أو الفقد |
|---|---|
| الهوية وبيانات الاتصال | ضرر خصوصية، استغلال، تمييز، تعريض طفل أو أسرة للخطر |
| البيانات الصحية والنمائية والتعليمية | وصم، قرارات غير صحيحة، أذى اجتماعي أو قانوني |
| الموافقة ونطاقها | معالجة غير مشروعة أو مشاركة خارج الغرض |
| بنود الاختبار ومفاتيح التصحيح والمعايير | إبطال أمن الاختبار ومخالفة حقوق الناشر |
| نتائج الجلسات وصلاحيتها | استنتاجات وتقارير غير صحيحة |
| قواعد المسار ومصادرها | توجيه الحالات إلى خطوات غير مناسبة |
| مؤهلات المستخدمين وتراخيصهم | تطبيق أدوات بواسطة غير المؤهلين |
| التقارير الموقعة | تزوير قرار مهني أو تعديل غير مكتشف |
| سجل التدقيق | إخفاء إساءة استخدام أو تعديل أو تسريب |
| مفاتيح التشفير وأسرار الخدمة | اختراق واسع للمنصة والنسخ الاحتياطية |

## 3. الجهات والخصوم المحتملون

- مستخدم خارجي يحاول الدخول إلى النظام.
- مقدم خدمة يحاول الوصول إلى حالة غير مكلف بها.
- موظف ذو صلاحية إدارية يحاول قراءة بيانات سريرية دون حاجة.
- مؤسسة تحاول الوصول إلى بيانات مؤسسة أخرى.
- حساب مهني مخترق.
- مستخدم مرخص لأداة يحاول نسخ موادها أو توزيعها.
- تكامل خارجي أو ناشر يتعرض للاختراق.
- خطأ برمجي أو إعداد خاطئ أو ترحيل قاعدة بيانات غير صحيح.
- موظف حسن النية يرفع ملفًا إلى خدمة خارجية غير معتمدة.
- نموذج ذكاء اصطناعي يعيد إنتاج بيانات حساسة أو يختلق استنتاجًا.
- مهاجم يغيّر قواعد المسار أو سجل الأدوات أو حالة الترخيص.

## 4. حدود الثقة

```text
جهاز مقدم الخدمة
  ↓ هوية موثقة + MFA
بوابة التطبيق وواجهة API
  ↓ سياق المؤسسة والمستخدم والحالة
خدمة الصلاحيات ومحرك المسارات
  ↓ مراجع معتمة فقط
قاعدة البيانات السريرية

مخزن الهوية المنفصل ← لا أسماء في قاعدة العمل السريري
حدود الناشر المرخصة ← لا مواد محمية في قاعدة المنصة
مخزن الوثائق المشفر ← تقارير وإثباتات مرخصة
سجل تدقيق غير قابل للتغيير ← نسخة مستقلة من أحداث المنصة
نظام النسخ الاحتياطي ← تشفير ومفاتيح منفصلة
```

كل انتقال عبر حد ثقة يحتاج مصادقة وتفويضًا وتسجيلًا والتحقق من سلامة البيانات.

## 5. التهديدات والضوابط

### 5.1 انتحال الهوية

**السيناريوهات**

- سرقة كلمة مرور مقدم خدمة.
- استخدام حساب موظف غادر المؤسسة.
- قبول ادعاء دور مهني من الواجهة أو الطلب مباشرة.
- إعادة استخدام جلسة أو رمز وصول مسروق.

**الضوابط**

- MFA إلزامي للحسابات المهنية والإدارية.
- هوية موحدة تدعم PKCE، ورموز قصيرة العمر، وتدوير رموز التحديث.
- استخراج المؤسسة والدور من مطالبات موثقة، لا من جسم الطلب.
- تعطيل الحساب فور انتهاء العلاقة أو الترخيص.
- إعادة تحقق عند التوقيع أو تصدير تقرير كامل.
- ربط الجلسة بمخاطر الجهاز والموقع والسلوك عند ملاءمة القانون.

### 5.2 العبث بالبيانات

**السيناريوهات**

- تعديل نتيجة أو حذف انحراف في التطبيق.
- تغيير مسار أو مصدر بعد استخدامه في حالة.
- تعديل تقرير موقع.
- الكتابة فوق تحديث مستخدم آخر.

**الضوابط**

- سجلات سريرية وإصدارات تقارير غير قابلة للتعديل.
- بصمة SHA-256 لتعريف المسار والتقرير والملفات المرجعية.
- ETag و`If-Match` للتحديثات القابلة للتغيير.
- قواعد ومسارات ذات إصدارات لا تتغير بعد التفعيل.
- فصل إنشاء التقرير عن توقيعه.
- سجل تدقيق متسلسل ببصمة الحدث السابق، ونسخة خارجية غير قابلة للتغيير.

### 5.3 إنكار الفعل

**السيناريوهات**

- إنكار فتح ملف أو تصديره.
- إنكار تعديل خطة أو تأكيد انتقال.
- إنكار توقيع تقرير أو مراجعة خطر.

**الضوابط**

- تسجيل المستخدم والمؤسسة والحالة والسبب ومعرّف الارتباط والوقت.
- توقيع تقارير ببصمة المحتوى ومرجع الترخيص المهني.
- تسجيل المحاولات المرفوضة، لا العمليات الناجحة فقط.
- مزامنة زمنية موثوقة وختم زمني عند الحاجة القانونية.
- منع مسؤولي النظام من حذف أحداث التدقيق.

### 5.4 كشف المعلومات

**السيناريوهات**

- استعلام عبر المؤسسة أو عبر حالة غير مكلف بها المستخدم.
- ظهور بيانات في سجل أخطاء أو أداة تحليلات.
- إرسال تقرير إلى بريد أو خدمة ذكاء اصطناعي غير معتمدة.
- نسخ بنود اختبار أو التقاط الشاشة.
- تسريب نسخة احتياطية أو ملف تصدير.

**الضوابط**

- فصل الهوية عن قاعدة العمل السريري.
- RLS على المؤسسة والتكليف بالحالة.
- أقل صلاحية، وحقول مخفية بحسب الدور.
- عدم تسجيل أجسام الطلبات الحساسة في السجلات التقنية.
- حظر أدوات التحليلات الخارجية في المسارات المهنية.
- روابط تنزيل قصيرة العمر، وتشفير ملفات التصدير، ووسمها باسم المستلم.
- منع تخزين مواد الناشر في المنصة إلا ضمن تكامل مرخص مستقل.
- تشفير النقل والتخزين والنسخ الاحتياطية بمفاتيح منفصلة.

### 5.5 حجب الخدمة

**السيناريوهات**

- إغراق تسجيل الدخول أو البحث أو إنشاء التقارير.
- إدخال ملفات ضخمة أو معقدة.
- فشل خدمة الهوية أو قاعدة البيانات أثناء جلسة.
- قفل شامل بسبب خطأ في قواعد السلامة.

**الضوابط**

- تحديد معدل الطلبات وحجم الملفات والمهل.
- قوائم انتظار للعمليات الثقيلة مع منع التكرار.
- حفظ تقدّم الجلسة بصورة آمنة عند السماح بذلك.
- وضع قراءة محدود للحالات الجارية أثناء بعض الأعطال، دون بدء جلسة أو توقيع تقرير.
- خطط استمرارية واستعادة مجربة، ومراقبة مؤشرات السعة.
- زر إيقاف مسار أو أداة محددة بدل إيقاف النظام كله عند الإمكان.

### 5.6 رفع الصلاحيات

**السيناريوهات**

- مقدم خدمة يغيّر دوره في الطلب.
- مسؤول مؤسسة يمنح نفسه دور مراجع سريري.
- مطبق أداة منتهية الصلاحية يبدأ جلسة.
- حساب دعم تقني يقرأ تقارير سريرية.

**الضوابط**

- أدوار خادم الهوية مع موافقة مزدوجة للأدوار الحساسة.
- تحقق حي من الترخيص والتدريب والمؤسسة والأداة قبل كل جلسة.
- فصل مسؤول النظام عن مسؤول الحوكمة السريرية.
- منع أدوار التطبيق من `BYPASSRLS` أو ملكية الجداول.
- مراجعة دورية للتكليفات والصلاحيات والوصول النادر أو الواسع.

## 6. أخطار السلامة السريرية

### H-01: تقرير نتيجة حاسمة من بيانات ناقصة

- **السبب:** تفسير القيمة المفقودة على أنها سلبية أو الانتقال رغم نقص مصدر مطلوب.
- **الضبط:** منطق ثلاثي `true/false/unknown`، قائمة حقول مفقودة، منع التقرير حتى المراجعة.
- **الاختبار:** حذف كل حقل مطلوب منفردًا والتأكد من عدم الوصول إلى التقرير.

### H-02: اعتماد أداة منفردة

- **السبب:** ارتفاع درجة مسح أو ملاحظة واحدة.
- **الضبط:** حد أدنى للمصادر والمجالات، وحظر انتقال جمع الأدلة مباشرة إلى التقرير.
- **الاختبار:** تقديم نتيجة أداة واحدة والتأكد من طلب أدلة إضافية.

### H-03: استخدام أداة دون ترخيص أو أهلية

- **السبب:** سجل قديم أو تجاوز واجهة المستخدم.
- **الضبط:** بوابة وصول خادمية قصيرة العمر قبل الجلسة، وإعادة التحقق عند البدء.
- **الاختبار:** انتهاء الترخيص أو التدريب أو حق الرقمنة بعد التخطيط وقبل الجلسة.

### H-04: تجاهل خطر عاجل

- **السبب:** استمرار المسار الروتيني بعد ظهور خطر.
- **الضبط:** حدث سلامة يضع الحالة في `safety_hold`، وإعادة الفحص عند معلومات جديدة.
- **الاختبار:** حقن مستوى `urgent` أو `emergency` في كل عقدة والتأكد من الإحالة العاجلة.

### H-05: خطأ في الهوية أو الحالة

- **السبب:** فتح نافذتين أو نسخ نتيجة لحالة أخرى.
- **الضبط:** شريط حالة ثابت، تأكيد معرّف داخلي قبل الاستيراد والتوقيع، منع عمليات جماعية عالية الخطر.
- **الاختبار:** محاولات إكمال جلسة بمعرّف حالة مختلف.

### H-06: تقرير غير صالح بسبب اللغة أو الحواس

- **السبب:** تطبيق أداة خارج لغتها أو دون تكييف صالح.
- **الضبط:** ملف لغة ووصول إلزامي، منع الدرجة المعيارية عند عدم الصلاحية، تسجيل الانحراف.
- **الاختبار:** ترجمة فورية لأداة معيارية والتأكد من التحويل إلى تفسير سردي محدود.

## 7. ضوابط الذكاء الاصطناعي

يسمح للذكاء الاصطناعي فقط داخل حدود معتمدة لأعمال مثل:

- تلخيص أدلة موجودة مع مراجعها.
- كشف الحقول الناقصة أو التناقضات ليراجعها المختص.
- إعداد مسودة صياغة غير موقعة.

ويمنع من:

- حساب أو تغيير درجات الاختبار.
- اختراع نقطة قطع أو معيار محلي.
- إصدار تشخيص أو أهلية أو توصية دوائية.
- الوصول إلى بنود أدوات محمية دون اتفاق ناشر صريح.
- تدريب نموذج على بيانات الحالات دون أساس قانوني وموافقة مستقلة وحوكمة بحثية.
- تنفيذ انتقال عالي الأثر دون تأكيد بشري.

يجب وسم كل نص مولد، وتخزين إصدار النموذج وتعليماته ومصادره، وإتاحة المقارنة والتعديل قبل أي توقيع.

## 8. متطلبات سجل التدقيق

يسجل على الأقل:

- تسجيل الدخول الفاشل والناجح وتغيير MFA.
- إنشاء الحساب وتعطيله وتغيير الدور.
- فتح الحالة والبحث عنها وتصديرها.
- تغيير الموافقة أو سحبها.
- إنشاء حدث سلامة ومراجعته وتسليمه.
- تخطيط أداة وفحص صلاحيتها وبدء الجلسة وإيقافها.
- إدخال نتيجة أو تعديل صلاحيتها أو تسجيل انحراف.
- تقييم عقدة مسار وتأكيد الانتقال.
- إنشاء مسودة تقرير وتعديلها وتوقيعها وسحبها.
- تغيير قواعد المسار أو سجل الأداة أو الترخيص.
- عمليات الإدارة والدعم الفني التي تؤثر في الوصول.

لا يسجل السجل كلمات مرور أو رموز وصول أو أجسام أدوات محمية أو بيانات أكثر من الحاجة.

## 9. اختبارات الأمن الدنيا

- اختبار فصل مؤسستين وحالتين ومستخدمين داخل المؤسسة نفسها.
- اختبار IDOR لجميع معرّفات الحالة والجلسة والتقرير والمصدر.
- اختبار انتهاء الرموز وإبطال الحساب والتدريب والترخيص.
- اختبار CSRF وXSS وحقن SQL وSSRF ورفع الملفات والروابط الموقعة.
- اختبار التزامن وإعادة إرسال الطلب ومفتاح منع التكرار.
- اختبار عدم تسرب البيانات في السجلات والأخطاء والتحليلات.
- اختبار النسخ الاحتياطي والاستعادة وتدوير المفاتيح.
- اختبار صلاحيات مسؤولي النظام والدعم والتدقيق.
- اختبار سلامة سلسلة التدقيق وكشف حذف أو إعادة ترتيب حدث.
- اختبار انقطاع الشبكة أو الخدمة أثناء جلسة أو توقيع.

## 10. شروط الإطلاق

لا يسمح ببيانات حقيقية قبل:

1. مراجعة نموذج التهديد بواسطة أمن مستقل.
2. اختبار اختراق وإغلاق جميع النتائج الحرجة والعالية.
3. DPIA وموافقة مسؤول حماية البيانات أو من يقوم مقامه.
4. تشغيل RLS واختبارات التكامل في بيئة مماثلة للإنتاج.
5. اعتماد خطة الاستجابة للحوادث والإبلاغ.
6. توثيق جميع المعالجات والاحتفاظ والمشاركة.
7. تفعيل المراقبة والتنبيه والاستعادة.
8. اعتماد كل تكامل ناشر وأداة بصورة منفصلة.
9. تجربة ببيانات صناعية، ثم تجربة ميدانية مراقبة بموافقة مستقلة.
